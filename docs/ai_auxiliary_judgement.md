# AI 辅助研判异步链路说明

## 核心原则

本系统的实时判断不要交给 AI，AI 只做异步辅助解释。

原因是 Gateway 串口数据是连续到达的，规则融合可以在毫秒级完成状态刷新，而本地 embedding 和大模型 API 都存在推理、排队、网络或服务启动延迟。若把实时生命微动判断交给 AI，会导致界面结论滞后，甚至出现旧数据覆盖新状态的问题。

因此，上位机应采用以下分工：

- 规则融合负责实时主判断。
- 本地 Jina embedding 负责异步模式匹配。
- 大模型 API 负责生成解释性辅助文本。
- UI 同时显示实时规则结论和带时间戳的 AI 辅助研判。

## 数据链路

完整链路如下：

```text
Gateway 串口数据
  ↓
DataManager 解析并缓存最近 5 秒多节点样本
  ↓
规则融合立即输出主判断
  ↓
异步任务取最近 5 秒数据生成结构化摘要
  ↓
本地 Jina embedding 服务输出向量
  ↓
本地模板相似度匹配输出模式标签
  ↓
可选大模型 API 生成解释性建议
  ↓
仪表盘显示 AI 辅助研判
```

这里的“最近 5 秒数据”不建议把原始 CSI 曲线直接交给模型，而是先转换为结构化摘要，例如：

```text
最近 5 秒共有 3 个节点参与。
node1 presence 均值 0.72，confidence 均值 0.81，motion 峰值 0.64，RSSI 稳定。
node2 presence 均值 0.69，confidence 均值 0.78，motion 峰值 0.58。
node3 presence 均值 0.18，confidence 均值 0.42。
规则融合结果：多节点疑似生命微动。
```

Jina 模型读取的是这类结构化文本摘要，并输出向量。向量只用于后台相似度计算，不直接展示给用户，也不直接作为最终判断。

## 输出位置

规则融合结果输出到仪表盘的“综合研判结果”区域，例如：

- `等待数据`
- `数据不足`
- `未检测到稳定微动`
- `疑似局部微动`
- `多节点疑似生命微动`

Jina embedding 输出的是向量，形式类似：

```json
{
  "embedding": [0.012, -0.083, 0.241]
}
```

该向量应保留在后台，用于和本地预设模板向量进行相似度比较。比较后得到可解释的模式结果，例如：

```json
{
  "top_matches": [
    {"label": "多节点疑似生命微动", "score": 0.86},
    {"label": "稳定微动特征", "score": 0.79}
  ]
}
```

大模型 API 不应直接接收大段原始向量，而应接收规则结论、节点摘要和相似模式结果，例如：

```text
规则判断：多节点疑似生命微动
参与节点：node1, node2, node3
触发节点：node1, node2
Jina 相似模式：多节点疑似生命微动，score=0.86
要求：输出一句谨慎的辅助研判，不要写确认生命。
```

大模型 API 的输出建议采用结构化结果，再由 UI 取其中的简短文本显示：

```json
{
  "conclusion": "多节点在最近窗口内形成一致微动响应",
  "reason": "node1 与 node2 同时出现较高 presence 和 confidence，具备交叉支持",
  "suggestion": "建议继续采集并重点观察该区域",
  "level": "observe"
}
```

仪表盘最终显示为：

```text
AI辅助研判：多节点在最近窗口内形成一致微动响应，建议继续采集并重点观察。
```

## 延迟处理

AI 辅助研判必须异步执行，不能阻塞串口接收、图表刷新和规则判断。

建议策略如下：

- 规则融合每次收到有效样本后立即刷新。
- 本地 Jina embedding 每 3 秒或状态发生明显变化时触发一次。
- 大模型 API 只在规则状态变化、Jina 相似模式变化或人工点击测试时触发。
- 每次 AI 请求必须记录分析窗口，例如 `12:00:05 - 12:00:10`。
- AI 返回后先检查当前数据时间，如果结果已经过期，不覆盖当前 UI。
- UI 可以显示 `AI辅助研判：分析中...`、`AI辅助研判：基于 4 秒前数据生成` 或 `AI辅助研判：上一轮分析已过期，正在更新`。

这样可以保证实时结论稳定可靠，同时保留 AI 的解释价值。

## 竞赛表述边界

对外展示时，应强调本系统是“规则融合主判断 + 双 AI 辅助解释”的结构，而不是把生命检测完全交给大模型。

推荐表述：

- `疑似生命微动`
- `多节点支持`
- `辅助研判`
- `建议继续采集观察`
- `当前数据不足，建议等待更多节点参与`

避免表述：

- `确认生命`
- `确认生命体征`
- `AI 已判定有人`
- `模型实时检测生命`

更稳妥的答辩说法是：

```text
系统实时结论由多节点规则融合产生，保证低延迟和可解释性。
AI 模块异步读取最近时间窗口内的结构化摘要，先由本地 Jina embedding 完成模式匹配，再由可选大模型 API 生成辅助解释。
AI 结果用于辅助展示和研判说明，不覆盖实时规则结论。
```

## 实现边界

后续实现时应保持以下边界：

- 不修改 Gateway 串口协议。
- 不让 AI 直接决定最终状态。
- 不把向量直接展示给用户。
- 不在 UI 线程里执行模型推理或网络请求。
- 没有 AI 配置或 API 不可用时，系统仍能依靠规则融合正常运行。
- AI 辅助区域只显示解释性建议和更新时间，不夸大检测能力。
