# AI 辅助研判功能说明

本文档说明 EchoGuard 上位机当前已经采用的 AI 辅助研判链路。核心链路为：

```text
最近 5 秒多节点历史样本
  -> 规则融合摘要文本
  -> Jina embedding
  -> 本地模板相似度匹配
  -> 云端大模型解释
  -> UI 显示 AI辅助研判文本
```

## 功能定位

AI 辅助研判不是实时主判断，也不直接判定“确认生命”。系统实时结论仍由多节点规则融合产生，AI 只在后台异步读取最近窗口摘要，生成解释性建议。

这种设计有三个目的：

- 保证串口接收、图表刷新和综合研判不被 AI 延迟阻塞。
- 让 AI 输出具备可解释性，而不是直接吃原始曲线或向量。
- 在 Jina 或云端模型不可用时，上位机仍可依靠规则融合正常工作。

## 构造组成

AI 辅助功能由四层组成：

- 实时数据层：`DataManager` 接收 Gateway 串口帧，维护节点状态和历史样本缓存。
- 规则摘要层：`build_detection_summary()` 从最近 5 秒多节点样本生成规则结论和结构化摘要文本。
- 本地语义匹配层：Jina embedding 服务将摘要文本和本地模板文本转换为向量，并计算相似度。
- 云端解释层：可选大模型 API 根据规则摘要和 Jina 相似模式生成一句谨慎的辅助研判文本。

当前 Jina 模型不直接读取原始 CSI 曲线数组，而是读取规则摘要文本。原始曲线先由上位机转换为 presence、confidence、motion、RSSI、参与节点、触发节点等可解释字段。

## 数据输入

上位机每收到有效 Gateway JSON 帧后，会把样本写入本次运行内历史缓存。AI 调度时只取最近 5 秒内的有效样本参与摘要构建。

摘要输入包含：

- 参与节点数量与节点名称，例如 `node1 / node2`
- 触发节点列表
- presence 均值
- confidence 均值
- motion 峰值
- RSSI 均值
- 规则融合状态
- 时间窗口信息

示例摘要：

```text
最近 5 秒共有 2 个节点参与。
参与节点：2；触发节点：node1, node2；时间窗口：最近 5 秒；建议继续采集观察
node1 样本 5 条，presence 均值 0.76，confidence 均值 0.81，motion 峰值 0.64，RSSI 均值 -52 dBm。
node2 样本 4 条，presence 均值 0.71，confidence 均值 0.79，motion 峰值 0.58，RSSI 均值 -55 dBm。
规则融合结果：多节点疑似生命微动。
```

## 处理链路

1. `DataManager` 每秒检查一次是否需要触发 AI 分析。
2. 若 AI 已启用、有有效参与节点、距离上次分析超过约 3 秒，且状态窗口有变化，则启动后台线程。
3. 后台线程调用 `run_ai_judgement()`。
4. `run_ai_judgement()` 先调用 Jina embedding，将当前摘要文本和本地模板文本一起向量化。
5. 本地计算当前摘要向量与模板向量的余弦相似度，得到 `top_matches`。
6. 若大模型 API 已启用，则把规则状态、参与节点、触发节点、时间窗口、Jina 相似模式和摘要文本发送给云端模型。
7. 云端模型返回一句中文辅助解释。
8. 上位机检查返回结果是否仍对应当前状态窗口，未过期才写入 UI。

## Jina embedding 作用

Jina embedding 的作用是把“当前窗口摘要”和“本地典型模式模板”映射到同一语义空间，用于本地相似度匹配。

当前本地模板包含类似：

- 多节点疑似生命微动
- 单节点局部扰动
- 未检测到稳定微动
- 数据不足

Jina 输出的原始向量不直接显示给用户，也不直接发送给云端大模型。系统只保留相似度匹配后的可解释结果，例如：

```json
{
  "top_matches": [
    {
      "label": "多节点疑似生命微动",
      "score": 0.86,
      "advice": "建议继续采集并重点观察触发节点区域"
    }
  ]
}
```

这样比把一长串向量数字交给大模型更稳定，也更容易在答辩时解释。

## 云端大模型作用

云端大模型只做解释性文本生成，不接管实时状态。

发送给大模型的信息包括：

- 规则判断
- 参与节点
- 触发节点
- 最近 5 秒时间窗口
- Jina 相似模式与分数
- 结构化摘要文本

大模型被约束为输出一句谨慎的中文建议，不能写“确认生命”。UI 最终显示格式为：

```text
AI辅助研判：多节点在最近窗口内同时触发，建议继续重点观察。
```

如果大模型 API 未启用或请求失败，系统退回本地 Jina 匹配文本；如果 Jina 也不可用，则显示规则回退文案。

## 异步与过期控制

AI 分析不是每帧运行，而是低频异步运行：

- 最短触发间隔约 3 秒。
- 同一状态窗口短时间内不重复请求。
- AI 运行期间不会阻塞串口、图表和实时规则结论。
- 每次请求携带 `state_key`、`window_start`、`window_end`。
- AI 返回后再次计算当前窗口，如果状态已变化，旧结果不会覆盖 UI。

因此 UI 上的主结论始终是实时规则结论，AI 区域只是辅助解释。

## 输出与回退

AI 状态写入快照中的 `ai` 字段，主要包含：

- `text`：显示到仪表盘的 AI 辅助研判文本
- `source`：`rule_fallback`、`local_jina` 或 `llm_api`
- `status`：当前 AI 状态，例如 `规则回退`、`本地 Jina 模式匹配完成`、`大模型辅助解释完成`
- `window_start` / `window_end`：本次 AI 分析对应的数据窗口
- `top_matches`：Jina 本地相似模式结果
- `error`：AI 失败原因

回退顺序为：

```text
云端大模型解释
  -> 本地 Jina 模式匹配文本
  -> 规则回退文本
```

没有 AI 配置、没有 Jina 服务、没有云端 API、网络异常时，上位机仍能继续显示规则融合结果。

## 与实时规则的边界

实时规则负责：

- `等待数据`
- `数据不足`
- `未检测到稳定微动`
- `疑似局部微动`
- `多节点疑似生命微动`

AI 辅助负责：

- 解释为什么当前窗口可能属于某类模式
- 给出继续采集、等待多节点支持、重点观察等建议
- 为竞赛展示提供更清晰的语言化说明

AI 不负责：

- 直接确认生命
- 直接修改串口协议
- 直接控制硬件
- 覆盖实时规则结论
- 根据过期窗口改写当前状态

## 部署关系

本地 Jina 使用 llama.cpp `llama-server` 提供 OpenAI 兼容 `/v1/embeddings` 接口。上位机支持在线部署、导入离线包和生成离线包，具体见 `docs/local_jina_deployment.md`。

云端大模型使用 OpenAI 兼容接口或智谱 GLM 官方接口。API Key 保存到本机用户配置，不提交到 GitHub。

## 答辩推荐表述

可以这样描述本功能：

```text
本系统采用“规则融合实时主判断 + 双 AI 异步辅助解释”的结构。
上位机先基于最近 5 秒多节点样本生成可解释摘要，再由本地 Jina embedding 完成模式相似度匹配。
若配置了云端大模型，系统会把规则摘要和相似模式交给大模型生成简短辅助建议。
AI 输出只用于解释和展示，不覆盖实时规则结论，也不宣称确认生命。
```
